## CSEE-Gen 2.0 系统需求文档

1. 系统目标实现从“年/多年”原始气象数据到“30天/可调天数”实验硬件指令的自动化转化。系统必须兼容降雨、日照、风速、降雪、冻融五大应力，并确保硬件执行的安全与机理的科学。

2. 核心逻辑架构系统分为三个解耦的阶段：
阶段一：全量应力扫描 (Global Stress Scanning)输入：全量小时级 CSV 数据（time, temperature, rain, snowfall, radiation, humidity）。降雨-日照识别：调用 RainEventExtractor 识别独立降雨事件。冻融循环识别：调用 compute_freeze_thaw_cycles 识别有效过零点事件。降雪总量统计：统计全年总降雪量 $V_{snow\_total}$ 供后续分配。
阶段二：等效映射与指令编排 (Mapping & Sequencing)雨热等效：基于能量守恒映射实验室日照强度。冰水等效：计算自然界总强度 $I_{env\_total}$ 与实验室单位强度 $I_{lab}$。确定实验室循环次数 $N$。均分补水：将 $V_{snow\_total}$ 均分到 $N$ 次循环的“补水步（Snow/Rain ON）”中。全局排序：将所有识别出的事件按“加权时间戳”混合排序，恢复自然应力节律。
阶段三：硬件级 CSV 导出 (Hardware-Ready Exporter)过渡步注入：在任何温度改变或模式切换前，自动插入 $3^\circ\text{C/min}$ 的调温缓冲步。全字段填充：确保输出 22 个字段，包括起止温度、起止日照强度、降雨/降雪标志位等。

3. 核心算法伪代码 (Pseudo-code)Pythonclass CSEE_Gen_v2:
    def __init__(self, target_days=30, ramp_rate=3.0):
        self.target_days = target_days
        self.ramp_rate = ramp_rate
        self.current_temp = 20.0  # 初始假设温度

    def process(self, raw_csv):
        # --- 步骤 1: 扫描全量应力 ---
        rain_events = RainExtractor.extract(raw_csv)
        ftc_events = FTCCalculator.compute_cycles(raw_csv)
        total_snow = raw_csv['snowfall'].sum()

        # --- 步骤 2: 等效计算 ---
        # 冻融折算
        n_ftc = calculate_mapping(ftc_events) 
        avg_snow_per_cycle = total_snow / n_ftc
        
        # --- 步骤 3: 构造指令序列 ---
        master_sequence = self.merge_and_sort(rain_events, n_ftc)

        # --- 步骤 4: CSV 生成与过渡注入 ---
        hardware_rows = []
        for event in master_sequence:
            # A. 检查温度变化，注入过渡步
            if event.target_temp != self.current_temp:
                ramp_step = self.create_ramp_step(self.current_temp, event.target_temp)
                hardware_rows.append(ramp_step)
                self.current_temp = event.target_temp

            # B. 注入正式实验步
            if event.type == "FREEZE_THAW":
                # 执行补水(雪)-冷冻-融化三部曲
                hardware_rows.extend(self.create_ftc_block(event, avg_snow_per_cycle))
            else:
                # 执行雨-光-干三部曲
                hardware_rows.extend(self.create_rain_solar_block(event))
        
        return save_to_csv(hardware_rows)
4. 关键规则约束 (Constraints)温度连续性（Continuity）：每一行的 起始温度 必须严格等于上一行的 终点温度。水光互斥（Mutex）：如果 日照控制标志 == ON，则 降雨/降雪量 必须强制为 0。角度联动（Solar Angle）：日照步必须根据运行时间，以 $5^\circ\text{min}$ 自动计算 起始角度 和 终点角度。动态爬坡（Dynamic Ramp）：过渡步时间按 abs(T_end - T_start) / 3.0 动态计算，最小取 1 分钟。5. 可行性与可解释性分析可行性：硬件对接：逻辑直接对应自动化导轨和分舱控制。计算量：所有逻辑均为线性扫描或简单公式，秒级输出。可解释性：机理一致性：通过 $I_{env}/I_{lab}$ 的强度折算，保证了累积损伤的一致性。水分闭环：通过“降雪量均分”确保了冻融阶段的有效含水率。
